# Telegram-бот «Ежедневник»

Многопользовательский Telegram-бот для планирования задач свободным текстом.
Пользователь пишет «завтра в 15 встреча с врачом» — бот парсит дату/время/задачу
через Gemini, сохраняет в БД, показывает список на день и шлёт напоминания.

## Стек

- **aiogram 3** — Telegram Bot API (async)
- **SQLAlchemy (async) + aiosqlite** — ORM поверх SQLite
- **Gemini API** — парсинг свободного текста в структуру
- **APScheduler** — периодический «тик» для напоминаний и дайджестов

## Ключевые решения

- **Часовые пояса:** per-user. Всё хранится в UTC, у каждого пользователя своя
  IANA-зона (`Europe/Moscow`), задаётся при онбординге и командой `/timezone`.
- **Напоминания:** за `N` минут до события (настраивается) + опциональный
  утренний дайджест со списком задач на день. Напоминания «ровно в момент» не шлём.
- **Повторяющиеся задачи:** поддерживаются (`daily` / `weekly` / `monthly`).
- **Действия над задачами:** создание, просмотр, редактирование, удаление.
- **Планировщик — модель «один тик»:** APScheduler запускает одну job раз в минуту.
  Она опрашивает БД и решает, какие напоминания созрели, кому слать дайджест и
  какие вхождения повторов наступили. Логика расписания живёт в данных, а не в
  планировщике: нет отдельного job-store, нет рассинхрона «задача ↔ job»,
  состояние переживает рестарт, а ядро легко тестировать как чистую функцию
  «что созрело к моменту T». Цена — минутная гранулярность (для ежедневника достаточно).

## Модели БД

### User

| Поле | Тип | Описание |
|------|-----|----------|
| `id` | int, PK | внутренний идентификатор |
| `telegram_id` | int, unique, indexed | ID пользователя в Telegram |
| `username` | str, nullable | @username (для удобства) |
| `timezone` | str | IANA-зона, напр. `Europe/Moscow`; дефолт из конфига |
| `digest_enabled` | bool | включён ли утренний дайджест |
| `digest_time` | str | локальное время дайджеста, напр. `"09:00"` |
| `default_lead_minutes` | int | за сколько минут напоминать по умолчанию (15) |
| `created_at` | datetime | момент регистрации |

### Task

| Поле | Тип | Описание |
|------|-----|----------|
| `id` | int, PK | идентификатор задачи |
| `user_id` | int, FK → User | владелец |
| `title` | str | что запланировано |
| `raw_text` | str | исходное сообщение (отладка/редактирование) |
| `due_at_utc` | datetime (UTC) | для разовых — момент события; для повторов — опорное время (берётся time-of-day) |
| `recurrence` | str | `none` \| `daily` \| `weekly` \| `monthly` |
| `recurrence_weekday` | int, nullable | 0–6, для `weekly` |
| `lead_minutes` | int, nullable | при NULL берётся `User.default_lead_minutes` |
| `last_reminded_on` | date, nullable | защита от повторной отправки в рамках polling |
| `created_at` | datetime | момент создания |

**Заметки по модели:**
- Повторяющиеся задачи **не материализуются отдельными строками** — вхождения
  вычисляются «на лету» в тике из `recurrence` + `due_at_utc`.
- Удаление — жёсткое (`DELETE`).
- `last_reminded_on` хранит дату последнего отправленного напоминания: для разовых
  гарантирует одну отправку, для повторов — одну отправку на вхождение в сутки.

| Команда | Назначение |
|---------|-----------|
| `/start` | регистрация + онбординг (выбор TZ) через FSM |
| `/help` | справка с примерами свободного текста |
| `/today` | задачи на сегодня |
| `/list <дата\|период>` | напр. `/list завтра`, `/list 20.06` |
| `/timezone` | установить часовой пояс |
| `/settings` | дайджест вкл/выкл, время дайджеста, дефолтный lead-time |
| `/cancel` | выход из текущего FSM-диалога |
| **свободный текст** | основной поток создания задачи (см. ниже) |

**Поток создания (свободный текст):**

```
«завтра в 15 встреча с врачом»
        │
        ▼
   Gemini парсит ──► превью с инлайн-кнопками:
                     [ ✅ Сохранить ]  [ ✏️ Изменить ]  [ ❌ Отмена ]
        │
        ▼
   сохранение в БД
```

**Инлайн-кнопки у задач в списке:** `✏️ изменить`, `🗑 удалить`.

## Поток данных

1. **Создание.** Текст → `gemini.py` (промпт включает часовой пояс пользователя и
   текущее datetime) → JSON `{title, datetime_local, recurrence, weekday}` →
   `parser.py` валидирует и переводит `local → UTC` → превью-подтверждение →
   `repo` сохраняет задачу.
2. **Тик (раз в минуту).** Для каждого пользователя:
   - ищем задачи, у которых `occurrence_utc − lead ≤ now` и `last_reminded_on != today` →
     отправляем напоминание, ставим `last_reminded_on = today`;
   - в локальное `digest_time` отправляем список задач на день (если `digest_enabled`);
   - вхождения повторов вычисляются на лету (без записи в БД).

## Обработка ошибок

- Gemini недоступен / вернул невалидный JSON / не извлёк дату → просим
  переформулировать, бот не падает.
- Неоднозначное или отсутствующее время → уточняющий вопрос (FSM) или дефолт.
- Неверная IANA-строка часового пояса → отказ с примерами корректных значений.
- Тик обёрнут в `try/except` на каждого пользователя — сбой одного не ломает рассылку остальным.
- Ошибки логируются; пользователь получает дружелюбное сообщение.

## Тестирование

TDD на чистых функциях:
- нормализация ответа парсера;
- расчёт вхождений повторяющихся задач;
- логика «что созрело к моменту `T`»;
- конвертация часовых поясов (`local ↔ UTC`).

`repo` тестируется на in-memory SQLite. Вызовы Gemini мокаются.

## План реализации по этапам

1. **Каркас и конфиг** — структура папок, `.env`, зависимости, бот-эхо-скелет.
2. **Слой БД** — модели, engine, `repo`, `create_all`, тесты.
3. **Парсинг свободного текста** — `gemini.py` + `parser.py` + превью/подтверждение
   (на этом этапе только разовые задачи).
4. **Просмотр** — `/today`, `/list`, форматирование списков.
5. **Планировщик** — тик, напоминания за `N` минут, guard от дублей.
6. **Повторы** — `recurrence` в парсере + логика вхождений + поддержка в тике.
7. **Редактирование/удаление** — инлайн-действия + FSM-редактирование.
8. **Настройки и онбординг** — `/timezone`, `/settings`, `/start`.
9. **Дайджест** — утренняя рассылка списка на день.
10. **Закалка** — обработка ошибок, логирование, тесты, README.
